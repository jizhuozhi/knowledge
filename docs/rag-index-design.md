## 文档类型定义

文档类型是决定索引策略的核心依据。我们需要一个统一的分类体系。

### 按内容性质分类

这是最核心的分类维度，决定了文档的知识形态和索引重点：

| 类型 | 特征 | 典型示例 | 索引重点 |
|-----|-----|---------|---------|
| **知识型文档** | 系统性、完整性、可学习 | 技术文档、教程、百科 | 向量索引+结构化分块 |
| **流程型文档** | 步骤性、顺序性、操作性 | SOP、部署手册、排查流程 | 步骤完整性+关联关系 |
| **数据型文档** | 结构化、数值密集、可统计 | 报表、配置表、监控数据 | 表格索引+精确查询 |
| **简报型文档** | 短文本、高密度、时效性 | 公告、告警、工单、会议纪要 | 关键词+时间衰减 |
| **经验型文档** | 问题-方案、可复用 | 故障案例、最佳实践 | 问题-方案索引+效果评估 |

### 按结构复杂度分类

| 类型 | 特征 | 典型示例 | 解析难度 |
|-----|-----|---------|---------|
| **结构化文档** | 固定格式、明确字段 | API文档、配置文档、FAQ | 低 |
| **半结构化文档** | 有标题章节、内容灵活 | 技术方案、需求文档、手册 | 中 |
| **非结构化文档** | 格式自由、内容松散 | 会议纪要、邮件、讨论记录 | 高 |
| **富媒体文档** | 文本+图表+代码混合 | 论文、技术报告、设计文档 | 高 |

### 按领域场景分类

| 场景 | 主要文档类型 | 特殊需求 |
|-----|------------|---------|
| **研发** | 技术方案、API文档、代码注释、论文 | 代码块处理、接口关联 |
| **产品** | PRD、竞品分析、用户手册 | 表格处理、需求追溯 |
| **运维** | SOP、故障手册、监控文档 | 步骤完整性、经验卡片 |
| **通用** | 会议纪要、公告、知识库 | 时效性、分类聚合 |

## 不同格式文档处理

不同文件格式有各自的解析难点和处理策略。

### Markdown / 纯文本

**特点**：产研场景最常用，结构清晰，解析简单

**处理要点**：
- 标题层级识别：通过 `#` 符号识别章节结构
- 代码块识别：通过 ` ``` ` 标记识别代码块，保持完整性
- 列表识别：有序/无序列表保持完整性
- 表格识别：Markdown表格转为结构化数据
- 链接和引用：提取链接目标，建立文档间关联

### Word文档

**特点**：产品/运维常用，格式多样，表格复杂

**处理要点**：
- 样式识别：通过字体、字号、加粗等样式识别标题层级
- 表格提取：保持表格结构，处理合并单元格
- 图片处理：提取图片和标题的关联
- 格式转换：转为中间格式便于后续处理

**表格处理的典型问题**：
- 合并单元格：需要展开还原
- 嵌套表格：需要识别并单独处理
- 表格标题：可能与表格物理位置分离

### PDF文档

**特点**：来源广泛，布局多样，解析难度相对较高

**处理要点**：
- 布局分析：识别单栏/多栏布局，按阅读顺序提取
- 元素识别：识别标题、段落、表格、图表、公式等
- 跨页处理：合并跨页元素，过滤页眉页脚
- 结构还原：识别章节边界，建立文档结构

**典型问题**：
- 多栏排版导致文本流错乱
- 表格/公式打断段落
- 图表与标题分离
- 跨页元素需合并

### Excel / CSV

**特点**：结构化数据，行列明确，可能多Sheet

**处理要点**：
- 多Sheet处理：识别各Sheet的语义关系
- 复杂表头：处理多行表头、合并表头
- 数据类型识别：文本、数值、日期等
- 空值处理：处理空单元格、缺失数据

**索引策略**：见"专项索引策略 - 表格索引"

### PPT

**特点**：演示文档，图文混排，内容碎片化

**处理要点**：
- 幻灯片粒度：每页作为一个基本单元
- 标题提取：提取幻灯片标题作为章节标识
- 图文关联：图片和文字的对应关系
- 演讲者备注：提取备注内容

### HTML

**特点**：网页内容采集的主要格式，结构丰富但噪声多

**处理要点**：
- DOM结构化解析：不使用正则去标签（丢失所有结构），而是通过DOM树遍历，将HTML结构化转换为Markdown
- 语义元素保留：标题(h1-h6)、段落(p)、列表(ul/ol)、表格(table)、代码块(pre/code)、链接(a)、图片(img)、引用(blockquote)、加粗/斜体等语义标签都应转换为对应的Markdown结构
- 噪声过滤：跳过script、style、noscript、head、meta等非可见元素
- 代码语言检测：从 `class="language-xxx"` 或 `class="lang-xxx"` 中提取代码块的编程语言
- 表格结构保留：将HTML table完整转换为Markdown管道符表格，处理表头分隔行
- 嵌套处理：正确处理嵌套列表、嵌套引用等复杂结构
- 空白折叠：非预格式化文本中折叠多余空白，预格式化文本（pre/code）保留原始格式

**典型问题**：
- 简单正则去标签会导致所有结构信息丢失，检索质量严重下降
- 复杂嵌套结构（如表格套列表、引用套代码块）需要递归状态追踪
- 某些网页大量使用div嵌套代替语义标签，需要优雅降级

**设计理念**：
HTML转Markdown的核心目标不是"去掉标签"，而是"保留结构语义"。Markdown作为中间表示格式，兼具人类可读性和结构化可解析性，是连接文档解析和分块索引的理想桥梁。

### 图片 / 扫描件

**特点**：需要OCR，可能含手写内容

**处理要点**：
- OCR识别：提取文字内容
- 布局还原：尽量还原原始布局
- 图表识别：识别图表并转为结构化数据

## 文档元数据

元数据是文档的"身份信息"，对检索和过滤至关重要。元数据分为两类：

### 硬性元数据

硬性元数据是文档固有的、可直接获取的信息，无需人工预定义：

**文件系统级**：
- 文件名、路径、格式
- 创建时间、修改时间
- 文件大小

**文档内置级**：
- Word/PDF 内置的元数据字段（标题、作者、主题）
- Markdown 的 frontmatter（YAML 格式的头部信息）
- 代码文件的注释头、package 信息

这些元数据直接从文件系统或文档格式中提取，不需要内容理解。

### 语义元数据

语义元数据需要从内容中理解和提取。采用泛化方案，不预定义要提取哪些字段：

**提取思路**：

让 LLM 阅读文档内容，自动识别并提取文档中的关键结构化信息。不限制 LLM 提取的字段类型，让它根据内容自行判断哪些信息值得提取。

**典型提取结果**：

| 文档类型 | 可能提取的元数据 |
|---------|-----------------|
| 技术方案 | 项目名、版本、负责人、技术栈、状态 |
| 会议纪要 | 会议主题、时间、参与人、决议事项 |
| 故障报告 | 故障时间、影响范围、根因、处理人 |
| 产品文档 | 产品名、版本、模块、功能点 |

**Prompt 设计要点**：

- 不在 Prompt 中硬编码要提取的字段
- 让 LLM 根据文档内容自行决定提取哪些信息
- 输出格式为 JSON，字段名和内容都由 LLM 生成
- 对于没有明显结构化信息的文档，允许输出空或少量字段

### 元数据的作用

**过滤检索**：
- 按时间范围筛选（"去年的技术文档"）
- 按作者筛选（"张三写的文档"）
- 按状态筛选（"进行中的项目"）

**增强检索**：
- 元数据作为检索条件的补充
- 元数据与内容检索结果融合

**关联分析**：
- 按项目聚合文档
- 按作者追踪贡献
- 按时间线组织内容

### 元数据存储

元数据采用 Schema-less 的存储方式：
- 不同文档的元数据字段不同
- 使用文档数据库（如 MongoDB）或搜索引擎（如 Elasticsearch）
- 支持动态字段和嵌套结构

## 文本索引

文本索引是最基础的索引类型，支持关键词精确匹配。

### 核心能力

**倒排索引**：建立词项到文档/分块的映射关系

**分词处理**：
- 中文：使用分词器（如jieba）切分
- 英文：基于空格和标点切分
- 专业术语：支持自定义词典

**查询能力**：
- 关键词精确匹配
- 短语匹配
- 前缀匹配
- 通配符匹配

### 适用场景

文本索引适用于：
- Brief类文档：关键词匹配是最主要的检索方式
- 专有名词查询：人名、地名、产品名、错误码
- 精确过滤：按标签、分类、时间范围过滤

### 与向量索引的关系

文本索引和向量索引是互补关系：

| 对比维度 | 文本索引 | 向量索引 |
|---------|---------|---------|
| 查询方式 | 关键词精确匹配 | 语义相似匹配 |
| 适用场景 | 专有名词、精确值 | 概念理解、泛化查询 |
| 召回特点 | 高精确、低召回 | 高召回、需排序 |
| 典型查询 | "ERROR 500" | "服务启动失败怎么办" |

在实际应用中，通常采用混合检索：
- 先用文本索引过滤精确匹配项
- 再用向量索引召回语义相关项
- 最后融合排序

## 向量索引

向量索引支持语义相似匹配，是RAG系统的核心索引。

### 向量化流程

```
文本内容 → Embedding模型 → 向量表示 → 向量数据库索引
```

### 向量索引策略

#### 单向量索引

每个文档/分块生成一个向量，最简单的方式。

**适用场景**：
- 短文本（<500字）
- 内容聚焦的文档

#### 多向量索引

同一文档/分块生成多个视角的向量：

| 向量类型 | 生成方式 | 作用 |
|---------|---------|-----|
| 内容向量 | 直接向量化原文 | 精确匹配原文内容 |
| 摘要向量 | 生成摘要后向量化 | 匹配文档主题 |
| 问题向量 | 生成潜在问题后向量化 | 匹配用户提问意图 |
| 标题向量 | 向量化标题 | 匹配文档主题关键词 |

**适用场景**：
- 长文档
- 内容复杂的文档
- 需要多角度检索的文档

#### 稀疏向量

结合关键词和向量，保留词汇信息：

- 稠密向量：捕获语义信息，维度低（如768维）
- 稀疏向量：保留词汇权重，维度高（词汇表大小）

**混合索引**：稠密向量 + 稀疏向量，兼顾语义和词汇匹配。

### 向量数据库选择

| 数据库 | 特点 | 适用场景 |
|-------|-----|---------|
| Milvus | 开源、高性能、支持多种索引 | 大规模生产环境 |
| Pinecone | 托管服务、易用 | 快速上线 |
| Weaviate | 内置向量化、支持多模态 | 需要一体化方案 |
| Qdrant | 轻量级、高性能 | 中小规模应用 |
| pgvector | PostgreSQL扩展 | 已有PG基础设施 |

### 检索参数

**top_k**：返回最相似的k个结果

**相似度阈值**：过滤相似度低于阈值的结果

**预过滤**：先按元数据过滤，再向量检索（如按时间范围、文档类型过滤）

## 图关系索引

图索引用于表示文档内和文档间的实体关系。

### 图索引的价值

文本索引和向量索引主要处理"内容匹配"，图索引处理"关系发现"：

- **实体关联**：找到与某个实体相关的所有信息
- **因果追溯**：追踪问题的根因
- **路径发现**：发现隐含的知识路径

### 图结构设计

#### 实体节点

| 节点类型 | 属性 | 示例 |
|---------|-----|-----|
| 文档 | id, 标题, 类型, 来源 | 技术方案文档 |
| 章节 | id, 标题, 层级, 位置 | 2.1 架构设计 |
| 实体 | id, 名称, 类型 | 用户服务(UserService) |
| 概念 | id, 名称, 定义 | 微服务 |

#### 关系类型

| 关系类型 | 起点 | 终点 | 语义 |
|---------|-----|-----|-----|
| CONTAINS | 文档 | 章节 | 包含 |
| REFERENCES | 文档 | 文档 | 引用 |
| DEPENDS_ON | 实体 | 实体 | 依赖 |
| IMPLEMENTS | 实体 | 概念 | 实现 |
| CAUSES | 实体 | 实体 | 导致 |

### 图索引适用场景

**技术文档**：接口调用关系、组件依赖关系

**运维文档**：故障因果链、服务依赖关系

**业务文档**：流程关联、组织关系

### 图谱构建方式

**方式一：规则抽取**

基于预定义规则抽取实体和关系：
- 正则匹配：提取特定格式的内容
- 模板匹配：如"A导致B"、"A依赖B"

**方式二：LLM抽取**

让LLM识别文本中的实体和关系：
- 提示LLM输出结构化的实体关系
- 灵活性高，但成本较高

**方式三：混合方式**

- 先用规则抽取明确的实体关系
- 再用LLM补充隐含的关系

### 图查询场景

**场景一：影响分析**

"修改服务A会影响哪些服务？"

→ 从服务A出发，遍历DEPENDS_ON关系

**场景二：根因追溯**

"服务不可用的根本原因是什么？"

→ 从故障节点出发，逆向遍历CAUSES关系

**场景三：知识路径发现**

"从问题A到解决方案B的知识路径是什么？"

→ 寻找两个节点间的最短路径

## 分块策略

分块是RAG系统的关键环节，直接影响检索质量和生成效果。

### 分块的核心目标

**语义完整性**：一个分块应该包含完整的语义单元

**粒度平衡**：
- 过大：包含无关信息，降低检索精度
- 过小：上下文不足，生成答案质量差

### 分块前预处理流水线

分块不应该直接在原始文本上操作。好的分块依赖于前置的结构化预处理。完整的流水线是：

```
原始文档（多种格式）
    │
    ▼
┌─────────────────┐
│  格式解析与转换   │  ← HTML→Markdown / Word→Markdown / 纯文本保持
│  (Document层)    │     核心目标：统一为结构化Markdown中间表示
└─────────────────┘
    │
    ▼
┌─────────────────┐
│  结构块解析       │  ← Markdown → Block序列
│  (Chunk层Phase1) │     识别heading/code_block/table/list/blockquote/paragraph/hr
└─────────────────┘
    │
    ▼
┌─────────────────┐
│  专项结构处理     │  ← 对table Block、code_block Block做特殊拆分
│  (Chunk层Phase2) │     表格→行级语义化 / 代码→语法感知拆分 / JSON→key级拆分
└─────────────────┘
    │
    ▼
┌─────────────────┐
│  策略分块         │  ← 在Block序列上按策略（语义/章节/滑动/父子）分块
│  (Chunk层Phase3) │     保证不跨Block边界
└─────────────────┘
    │
    ▼
Chunk序列 → 向量化 → 索引
```

**设计理念**：

1. **Markdown作为统一中间表示**：无论原始格式是HTML、Word、纯文本还是Markdown，都先转换为Markdown。Markdown兼具人类可读性和结构化可解析性，是连接"文档解析"和"分块索引"的理想桥梁。

2. **结构解析先于分块**：传统分块直接在文本上按字符数切割，对文档结构完全无感知。先将文本解析为结构块（Block），再在Block边界内分块，可以从根本上避免"从表格中间切断"、"从代码块中间切断"等问题。

3. **专项处理与通用分块分离**：表格行级语义化、代码语法感知拆分等专项处理在Phase 2完成，不与Phase 3的通用分块策略耦合。这意味着可以独立演进专项处理能力，而不影响通用分块逻辑。

4. **SpecialProcessing字段的消费**：索引策略推理输出的`SpecialProcessing`字段（如`table_aware`）在Phase 2被消费。这实现了"策略推理"→"策略执行"的闭环：LLM决定需要什么专项处理，预处理流水线执行相应的处理逻辑。

### 分块策略类型

#### 固定长度分块

按固定字符数或Token数切分。

**优点**：简单直接，易于实现

**缺点**：
- 可能切断语义单元
- 不考虑内容结构

**适用场景**：内容连续、结构简单的文档

#### 语义分块

按语义边界切分，通常基于语义相似度判断边界。

**方法**：
- 计算相邻句子的语义相似度
- 相似度低的边界作为分块点

**优点**：保持语义完整性

**缺点**：计算成本较高

**适用场景**：通用文档，内容多样化的场景

#### 结构化分块

按文档结构切分（章节、段落、列表项等）。

**章节分块**：每个章节作为一个分块

**段落分块**：每个段落作为一个分块

**列表分块**：每个列表项作为一个分块

**优点**：保持结构完整性

**缺点**：可能产生过大的分块

**适用场景**：有明确结构的文档（技术文档、论文）

#### 父子分块

同时生成大块和小块：
- 小块（128-256 tokens）：用于精确检索
- 大块（512-1024 tokens）：用于生成回答

**检索流程**：
1. 用小块进行检索，找到匹配的小块
2. 找到小块对应的父块
3. 用父块作为上下文生成答案

**优点**：兼顾检索精度和上下文完整性

**缺点**：存储成本翻倍

**适用场景**：长文档、内容复杂的文档

#### 滑动窗口分块

按固定窗口切分，窗口之间有重叠。

**优点**：保证上下文连续性

**缺点**：产生冗余

**适用场景**：连续性强的内容（代码、流程描述）

#### Markdown结构感知分块

传统分块（固定长度、语义分块、滑动窗口）都在"纯文本"层面操作，不感知Markdown的结构元素。结构感知分块的核心理念是：**先解析结构，再在结构边界内分块**。

**方法**：

将Markdown文本解析为结构块（Block）序列，每个Block有明确的类型：

| Block类型 | 识别方式 | 示例 |
|----------|---------|-----|
| heading | `#` 开头 | `## 架构设计` |
| code_block | ` ``` ` 围栏 | Go/Python/JSON代码块 |
| table | `|` 管道符行 | Markdown表格 |
| list | `-`/`*`/`数字.` 开头 | 有序/无序列表 |
| blockquote | `>` 开头 | 引用块 |
| paragraph | 普通文本段落 | 正文段落 |
| hr | `---`/`***`/`___` | 水平分隔线 |

**核心保证**：
- 表格不会被从中间切断——整个表格作为一个不可分割的Block
- 代码块不会被从中间切断——完整代码块是一个Block
- 列表不会被从中间切断——连续列表项构成一个Block
- 标题与其下属内容关联——标题Block携带层级信息，用于章节分块

**分块流程**：

```
Markdown文本
    │
    ▼
解析为Block序列  ← Phase 1：结构解析
    │
    ▼
特殊Block预处理  ← Phase 2：表格/代码块特殊处理（见"专项索引策略"）
    │
    ▼
在Block边界内分块 ← Phase 3：按策略（语义/章节/滑动窗口/父子）分块
```

**与其他策略的关系**：

结构感知不是一种独立的分块策略，而是**所有分块策略的前置层**。它提供结构边界信息，具体的分块仍然由语义分块、章节分块、滑动窗口、父子分块等策略完成：

- **结构感知 + 语义分块**：在Block序列上做语义相似度判断，但不跨Block边界合并
- **结构感知 + 章节分块**：以heading Block为分割点，每个章节的Block序列作为一个chunk
- **结构感知 + 滑动窗口**：窗口在Block级别滑动，而非字符级别
- **结构感知 + 父子分块**：父块和子块都尊重Block边界

**优点**：
- 保证表格、代码块、列表等结构元素的完整性
- chunk的语义边界更自然
- 为后续的表格/代码特殊处理提供结构化输入

**缺点**：
- 需要Markdown格式输入（其他格式需先转换为Markdown）
- 解析逻辑比纯文本分块复杂

**适用场景**：所有经过Markdown格式化的文档（Markdown原生文档、HTML转Markdown、Word转Markdown等）

### 分块边界处理

分块时遇到特殊元素，需要特殊处理：

#### 表格

**问题**：表格打断段落，上下文丢失

**策略**：
- 表格独立成块
- 保持表格与上下文的关联（元数据标记）
- 检索时一并返回关联内容

#### 代码块

**问题**：代码块被截断，语法结构破坏

**策略**：
- 小代码块（≤chunkSize）不被切分，整体作为一个分块单位
- 大代码块按语法边界切分（函数定义、类定义、JSON顶级key等），详见"专项索引策略 - 代码块/JSON语法感知索引"
- 元数据标记编程语言、符号名称等信息
- 切分后每个片段是语法上完整的、可独立理解的

#### 列表/步骤

**问题**：步骤跨块，完整性丢失

**策略**：
- 一个步骤/条目不跨块
- 保持有序步骤的顺序性
- 检索时可能返回完整步骤序列

### 分块参数

| 参数 | 说明 | 典型值 |
|-----|-----|-------|
| 分块大小 | 每个分块的Token数 | 256-512 |
| 重叠大小 | 相邻分块的重叠Token数 | 50-100 |
| 最小块大小 | 过小的块合并到相邻块 | 100 |

### 分块策略选择

根据文档类型和内容特征选择分块策略：

| 文档类型 | 推荐策略 | 分块大小 |
|---------|---------|---------|
| 技术文档 | 章节分块 | 300-500 |
| 论文 | 语义分块 | 400-600 |
| API文档 | 按接口分块 | 200-300 |
| 流程文档 | 列表分块 | 按步骤 |
| Brief | 不分块 | 原文 |
| 长文档 | 父子分块 | 小:200, 大:800 |

## 长文档AI摘要索引

对于长文档，分块索引存在局限，需要额外的摘要索引层。

### 长文档索引的问题

**分块数量多**：检索时需要遍历大量分块，效率低

**全局视角缺失**：难以理解文档整体内容

**分块间关联弱**：容易检索到碎片化信息

### 解决方案：AI摘要索引

为长文档生成AI摘要，作为额外的索引层。

#### 摘要层级

**文档级摘要**：概括全文核心内容（200字左右）

**章节级摘要**：每个章节的一句话总结

**关键信息提取**：
- 关键观点
- 关键实体
- 关键关系

#### 检索流程

```
用户查询
   │
   ▼
摘要向量检索 ──> 定位相关文档
   │
   ▼
在候选文档的分块中精细检索
   │
   ▼
融合排序返回
```

**融合策略**：摘要分数 × 0.3 + 分块分数 × 0.7

### 触发条件

满足以下任一条件时生成摘要：

| 条件 | 说明 |
|-----|-----|
| 文档长度 > 5000字 | 超长文档必须有摘要 |
| 文档长度 > 2000字 且 有多层章节结构 | 复杂结构需要导航 |
| 文档长度 > 3000字 且 包含表格/图表 | 富媒体需要概览 |

### 摘要类型选择

| 文档特征 | 摘要类型 | 内容 |
|---------|---------|-----|
| 长度 > 10000字 | 完整摘要 | 文档级 + 章节级 + 关键信息 |
| 长度 5000-10000字 | 部分摘要 | 文档级摘要 |
| 有章节结构 | 章节摘要 | 仅章节级摘要 |

## 专项索引策略

某些文档类型有特殊的索引需求，需要专门的索引策略。

### 表格索引

表格是结构化数据，直接转为文本向量化会丢失结构信息。

#### 表格索引策略

**策略一：摘要向量化**

将整个表格生成一段自然语言摘要，然后向量化。

摘要内容：
- 表格标题
- 包含的列名
- 行数统计
- 数值列的统计信息（总和、平均值等）

适用场景：小表格（≤10行），用于粗粒度匹配

**策略二：行级向量化**

将每一行转换为自然语言描述，然后单独向量化。

转换方式：将"姓名=张三，年龄=25，部门=研发"转换为"姓名为张三，年龄为25，部门为研发"

适用场景：中等表格（11-100行），用于精确行匹配

**策略三：列级索引**

对每一列建立倒排索引，支持精确值查询。

适用场景：大表格（>100行），用于列值过滤

**策略四：图谱化**

将表格结构转为图数据库中的节点和关系：
- 表格节点
- 列节点
- 行节点
- 单元格节点

适用场景：复杂表格，需要关联查询

#### 策略选择

| 表格大小 | 摘要向量化 | 行级向量化 | 列索引 | 图索引 |
|---------|-----------|-----------|--------|--------|
| 小（≤10行） | ✓ | ✗ | ✗ | ✗ |
| 中（11-100行） | ✓ | ✓ | ✗ | ✗ |
| 大（>100行） | ✓ | ✓ | ✓ | ✓ |

#### 表格检索流程

1. 摘要匹配：用摘要向量粗筛相关表格
2. 行匹配：在候选表格的行向量中精细匹配
3. 列值匹配：对特定列值进行精确查询
4. 融合排序：综合各路结果返回

### 代码块/JSON语法感知索引

代码块和JSON是高度结构化的内容，直接按字符数切分会破坏语法结构，导致生成时无法还原完整可用的代码。

#### 设计理念

代码和JSON有明确的语法树结构，应该按语法边界而非字符边界切分：
- **函数/类定义**是代码的自然语义单元
- **JSON的顶级key**是数据的自然语义单元
- 切分后的每个chunk应该是语法上完整的、可独立理解的片段

#### 分块策略

**小代码块（≤chunkSize）**：不做拆分，整体作为一个chunk。

**大代码块（>chunkSize）**：按语言类型选择切割策略：

| 语言 | 切割边界 | 策略 |
|-----|---------|-----|
| **JSON对象** | 顶级key | 每个顶级key-value对作为独立chunk，附带JSON路径前缀 |
| **JSON数组** | 数组元素 | 每N个元素（默认10个）为一批，附带数组上下文描述 |
| **Go** | `func`/`type`/`var`/`const` | 按函数、类型定义、变量声明边界切割 |
| **Python** | `class`/`def`/`async def` | 按类定义、函数定义边界切割 |
| **JavaScript/TypeScript** | `function`/`class`/`const`/`export` | 按函数、类、导出声明边界切割 |
| **未知/其他语言** | 空行 | 先尝试JSON检测，再按空行分隔的逻辑段落切割 |

**JSON切割示例**：

原始JSON（一个大配置文件）：
```json
{
  "database": { "host": "localhost", "port": 5432, ... },
  "cache": { "type": "redis", "ttl": 3600, ... },
  "logging": { "level": "info", "format": "json", ... }
}
```

切割为3个chunk：
- Chunk 1: `[JSON key: database] {"host": "localhost", "port": 5432, ...}`
- Chunk 2: `[JSON key: cache] {"type": "redis", "ttl": 3600, ...}`
- Chunk 3: `[JSON key: logging] {"level": "info", "format": "json", ...}`

**代码切割示例**：

原始Go代码（一个service文件）：
```go
func (s *Service) Create(...) error { ... }
func (s *Service) Update(...) error { ... }
func (s *Service) Delete(...) error { ... }
```

切割为3个chunk，每个函数定义是独立chunk。

#### Chunk元数据

语法感知分块生成的chunk应携带额外元数据：

| 元数据 | 说明 | 示例 |
|--------|-----|------|
| chunk_type | 分块类型 | `code_function`/`code_class`/`json_key`/`json_array_batch` |
| language | 编程语言 | `go`/`python`/`json`/`javascript` |
| symbol_name | 符号名称（如函数名） | `Create`/`UserService` |
| json_path | JSON路径 | `database`/`cache.redis` |

#### 策略选择

| 代码块大小 | 处理方式 |
|-----------|---------|
| ≤chunkSize | 整体保留，不拆分 |
| >chunkSize，已知语言 | 按语法边界拆分 |
| >chunkSize，未知语言 | 尝试JSON检测 → 按空行逻辑段落拆分 |

#### 与表格索引的统一模式

代码块索引和表格索引遵循相同的设计模式：

```
结构化内容（表格/代码块）
    │
    ▼
判断大小 → 小则整体保留
    │
    ▼ 需要拆分
按结构边界拆分（表头行/函数定义/JSON key）
    │
    ▼
每个片段独立成chunk，携带结构元数据
```

### Brief索引

Brief是短文本、高密度信息的文档类型。

#### Brief特征

- **长度**：10-500字符
- **特点**：信息密度高、时效性强、数量大
- **时效性**：需要时间衰减和过期管理
- **聚合需求**：强

#### Brief类型

| 类型 | 示例 |
|-----|-----|
| 业务类 | 公告通知、新闻简报、会议摘要 |
| 产品类 | 产品简介、功能更新、发布说明 |
| 运维类 | 故障日志、告警信息、系统消息 |
| 用户类 | 用户反馈、工单、评论 |

#### 索引策略

**文本索引（权重最高）**

关键词精确匹配是Brief最主要的检索方式。需要：
- 高效的倒排索引
- 支持时间范围过滤
- 支持标签过滤

**向量索引（需内容增强）**

短文本直接向量化效果差，需要增强语义信息。

增强方式：
- 添加类型信息
- 添加标签信息
- 添加元数据

示例：
- 原始内容："CPU使用率超过90%"
- 增强内容："CPU使用率超过90% | 类型：alert_message | 标签：severity:critical, component:api-server"

**时间索引**

- 按时间排序存储
- 支持过期自动清理
- 时间衰减权重

**标签索引**

- 按类型建立索引集合
- 按标签建立索引集合
- 支持多标签聚合查询

#### 时间衰减

越新的Brief权重越高：

```
衰减因子 = exp(-年龄天数 / 半衰期天数)
最终分数 = 基础检索分数 × 衰减因子
```

典型半衰期：30天

### 经验卡片索引

经验卡片是可复用的、结构化的知识单元，记录问题解决方案和最佳实践。

#### 核心字段

**问题场景**：
- 问题关键词
- 问题描述
- 症状表现
- 错误信息

**解决方案**：
- 解决方案描述
- 解决步骤
- 代码片段
- 命令行

**适用范围**：
- 技术栈
- 组件
- 场景
- 版本范围

**效果评估**：
- 有效性评分（0-5）
- 验证次数
- 成功率
- 用户反馈

#### 索引策略

**问题关键词索引**

对问题关键词建立倒排索引，支持关键词精确匹配。

特别处理错误信息：
- 提取错误码（如ERROR 500）
- 提取异常类名（如NullPointerException）
- 提取HTTP状态码

**解决方案索引**

对解决方案中的关键元素建立索引：
- 命令索引
- 技术栈索引
- 组件索引

**向量索引**

分别建立问题和解决方案的向量索引：
- 问题向量：问题描述+症状+错误信息
- 解决方案向量：解决方案+步骤+代码

#### 检索策略

多路召回 + 效果排序：

1. 错误信息精确匹配（权重0.5）：对错误码、异常类名精确匹配
2. 问题关键词匹配（权重0.3）：对问题关键词倒排检索
3. 语义相似匹配（权重0.2）：对问题向量进行相似检索
4. 效果评分加成：考虑有效性评分、成功率、使用频率

#### 效果评估闭环

用户使用经验卡片后，记录反馈：
- 是否有帮助
- 是否解决问题
- 评论

根据反馈更新：
- 验证次数+1
- 更新成功率
- 更新有效性评分

## 索引策略推理

不同文档应该采用什么索引策略？这是索引决策的核心问题。

### 传统方案的问题

**人工枚举规则**：
- 需要持续维护
- 新文档类型需要新增规则
- 泛化能力差

**ML模型分类**：
- 需要训练数据
- 没有精力单独训练模型

### 解决方案：LLM驱动策略推理

采用LLM直接推理索引策略：
- 不维护规则和ML模型
- 把策略知识编码在Few-Shot Prompt中
- LLM根据文档特征直接输出索引配置

### 推理流程

```
文档输入
   │
   ▼
特征提取（轻量级：长度、结构、关键词）
   │
   ▼
缓存查询 ──> 命中则直接返回缓存策略
   │
   ▼ 未命中
Few-Shot LLM推理
   │
   ▼
输出JSON格式的索引配置
   │
   ▼
执行索引
```

### 策略选项

**分块策略**：
| 策略 | 说明 | 适用场景 |
|-----|-----|---------|
| none | 不分块 | Brief、短文本 |
| semantic | 语义分块 | 通用文档 |
| by_section | 按章节分块 | 技术文档 |
| sliding | 滑动窗口 | 连续性强的内容 |
| parent_child | 父子分块 | 精确检索+完整上下文 |

**特殊处理**：
| 处理方式 | 说明 | 适用场景 |
|---------|-----|---------|
| table_aware | 表格感知索引 | 含表格文档 |
| brief_mode | Brief模式（时间衰减） | 短时效内容 |
| experience_card | 经验卡片提取 | 故障排查、最佳实践 |
| ai_summary | AI摘要生成 | 长文档 |

### Few-Shot Prompt示例

Prompt中包含若干示例，展示如何根据文档特征选择策略：

**示例1：技术API文档**
- 文档特征：长度3200字，有代码块，有接口定义，结构清晰
- 推荐策略：按章节分块，分块大小300字，启用图索引
- 理由：API文档按接口分块更精准，图索引用于关联接口调用关系

**示例2：故障排查手册**
- 文档特征：长度5800字，有步骤列表，有错误码，有解决方案
- 推荐策略：语义分块，启用经验卡片提取
- 理由：故障排查内容适合提取为经验卡片，语义分块保证问题-方案完整性

**示例3：产品需求文档**
- 文档特征：长度15000字，多层章节，有表格，有流程图描述
- 推荐策略：父子分块，启用AI摘要
- 理由：长文档需要AI摘要辅助定位，父子分块兼顾检索精度和上下文完整性

### 缓存策略

为避免每次索引都调用LLM，设计缓存机制：
- 对文档特征进行分桶哈希（如长度按1k分桶）
- 相似特征的文档返回相同策略
- 采用LRU淘汰策略管理缓存大小

## 增量更新机制

知识库内容会持续更新，需要高效的增量更新机制。

### 更新类型

**新增文档**：
- 解析文档
- 分类并生成索引策略
- 执行索引

**修改文档**：
- 删除旧索引
- 作为新增文档处理

**删除文档**：
- 从所有索引中删除对应数据
- 清理关联的分块、摘要等

### 设计思路

采用队列化异步更新：
- 文本索引、向量索引、图索引分别维护独立队列
- 后台Worker并行消费队列
- 更新操作入队后立即返回，不阻塞用户请求

## LLM优先与演进路径

### 当前阶段的设计取舍

本文档默认的前提是：团队希望先把知识库和Agent能力搭起来，但没有足够的算法资源去单独训练文档分类模型、索引策略模型，或长期维护复杂规则系统。在这种前提下，采用 LLM first 的方式进行文档理解和索引策略推理，是更泛化、也更经济实惠的做法。

这样做的价值在于：
- 不需要前期建设大量训练数据
- 不需要为每类文档单独维护规则
- 能更快覆盖新文档类型和长尾场景
- 能先支撑一个可用的知识库和Agent系统

### LLM first, but not LLM only

采用 LLM 优先，不等于系统只能依赖 LLM。更合理的方式是：让 LLM 负责高语义、难枚举的理解和策略判断，让系统负责稳定、可控的索引执行。

具体来说：
- 文档基础特征提取仍然由确定性流程完成，如长度、格式、标题层级、表格数量、代码块数量等
- 分块、向量化、倒排建索引、图构建等底层执行流程仍然是可控的工程能力
- LLM 输出的索引策略应落在系统支持的有限配置集合内，而不是任意生成执行逻辑
- 对高频文档特征可以做缓存，对异常场景可以回退到通用默认策略

### 有专业算法团队时的增强方向

如果后续有专业的检索/算法团队投入，可以在不推翻当前架构的前提下，逐步增加更可控的能力：
- 更稳定的文档理解与内容分类
- 更细粒度的索引策略选择
- 更专门的表格、流程、故障类文档识别
- 更可控的实体抽取、关系抽取与结构化处理
- 更系统的效果评估与策略迭代能力

### 演进原则

推荐的演进方式是：
- 第一阶段：以 LLM 优先驱动文档理解和索引策略推理，快速完成系统搭建
- 第二阶段：通过日志和使用反馈识别高频文档类型与高价值场景
- 第三阶段：对模式稳定、价值高的场景逐步引入规则、轻量模型或专项算法
- 第四阶段：形成“专用能力处理高频核心场景，LLM 处理长尾泛化场景”的混合体系

## 总结

### 文档类型与索引策略映射

| 文档类型 | 文本索引 | 向量索引 | 图索引 | 分块策略 | 特殊处理 |
|---------|---------|---------|--------|---------|---------|
| 知识型文档 | ✓ | ✓ | 可选 | 章节分块 | AI摘要（长文档） |
| 流程型文档 | ✓ | ✓ | ✓ | 步骤分块 | 保持顺序性 |
| 数据型文档 | ✓ | 可选 | 可选 | 表格策略 | 多维度索引 |
| 简报型文档 | ✓ | 增强 | ✗ | 不分块 | 时间衰减 |
| 经验型文档 | ✓ | ✓ | ✓ | 语义分块 | 经验卡片提取 |

### 格式与解析难度

| 格式 | 解析难度 | 关键问题 |
|-----|---------|---------|
| Markdown/纯文本 | 低 | 代码块边界、表格/列表完整性 |
| HTML | 低-中 | DOM结构化转换、噪声过滤、嵌套处理 |
| Word | 中 | 表格、样式识别 |
| Excel/CSV | 低 | 多Sheet、复杂表头 |
| PDF | 中-高 | 布局分析、阅读顺序 |
| PPT | 中 | 图文关联 |
| 图片/扫描件 | 高 | OCR、布局还原 |

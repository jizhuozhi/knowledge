## 检索流程概览

在线检索是将用户查询转换为相关文档的过程，核心流程：

```
用户查询
    │
    ▼
┌─────────────┐
│  查询理解    │  ← 理解意图、提取关键信息
└─────────────┘
    │
    ▼
┌─────────────┐
│  检索策略    │  ← 选择召回策略
└─────────────┘
    │
    ▼
┌─────────────┐
│  多路召回    │  ← 文本/向量/图多路并行
└─────────────┘
    │
    ▼
┌─────────────┐
│  结果融合    │  ← 合并、去重、排序
└─────────────┘
    │
    ▼
┌─────────────┐
│  重排序      │  ← 精排、过滤
└─────────────┘
    │
    ▼
返回结果
```

### 设计原则

**准确性优先**：召回的结果必须与查询相关

**效率平衡**：在保证准确性的前提下，控制检索耗时

**泛化能力**：不针对特定查询类型硬编码规则，让模型自适应

## 查询理解

查询理解是检索的第一步，目的是理解用户真正想要什么。

### 查询类型

用户的查询意图多样，但不需要预定义所有类型：

| 查询特点 | 示例 | 检索重点 |
|---------|-----|---------|
| 精确查询 | "ERROR 500 怎么解决" | 关键词匹配优先 |
| 概念查询 | "微服务架构怎么设计" | 语义匹配优先 |
| 对比查询 | "Redis 和 Memcached 区别" | 多实体关联 |
| 流程查询 | "如何部署服务" | 步骤完整性 |
| 探索查询 | "最近有什么技术更新" | 时间相关性 |

### 查询理解方案

采用 LLM 进行查询理解，不预定义规则：

**理解内容**：
- 查询意图：用户想做什么
- 关键实体：查询中提到的重要概念
- 时间意图：是否有时效性要求
- 过滤条件：是否隐含限定条件

**输出格式**：

让 LLM 输出结构化的理解结果，字段由 LLM 自行判断，不强制预定义。

**示例**：

查询："上周张三提交的技术方案"

LLM 理解结果：
- 时间范围：上周
- 作者：张三
- 文档类型：技术方案
- 检索目标：查找特定文档

查询："服务启动失败怎么办"

LLM 理解结果：
- 问题类型：故障排查
- 检索目标：解决方案
- 适合用经验卡片索引

### 查询改写

用户查询可能表述不完整或模糊，需要改写：

**改写场景**：
- 补充上下文：将"它怎么配置"改写为具体对象
- 扩展同义词：将"DB"扩展为"数据库"
- 纠正错误：修正拼写或表述错误

**改写方式**：

由 LLM 根据查询内容判断是否需要改写，不预设改写规则。

## 查询路由

不是所有查询都应该走知识库检索。根据查询特点，需要路由到不同的信息源。

### 信息源类型

**知识库检索**：
- 适用：内部知识、历史文档、已沉淀的经验
- 特点：信息可靠、可追溯、但可能有延迟
- 来源：内部文档库、知识库、Wiki

**外部搜索**：
- 适用：实时信息、行业动态、公开知识
- 特点：信息新、覆盖广、但需验证
- 来源：Web 搜索、API 数据源

**混合检索**：
- 适用：需要结合内部知识和外部信息
- 特点：内外结合、信息互补
- 方式：并行检索后融合

### 路由决策

由 LLM 判断查询应该走哪个信息源：

**走外部搜索的特征**：
- 强调时效性（"最近"、"最新"、"当前"）
- 外部公开信息（行业动态、市场价格、公开新闻）
- 知识库不太可能覆盖的内容

**走知识库检索的特征**：
- 内部知识（技术方案、内部流程、历史故障）
- 已沉淀的经验和文档
- 不强调时效性

**走混合检索的特征**：
- 需要内外信息结合
- 对比分析（内部做法 vs 行业实践）

### 路由方案

采用 LLM 进行路由决策，不预设硬编码规则：

**决策输入**：
- 用户查询
- 可用的信息源列表

**决策输出**：
- 目标信息源（知识库/外部/混合）
- 路由理由（为什么选择这个信息源）

**示例**：

| 查询 | 路由决策 | 理由 |
|-----|---------|-----|
| "最近有什么技术更新" | 外部搜索 | 强调时效性，需要最新信息 |
| "Redis 集群怎么部署" | 知识库检索 | 技术方案类，知识库已有沉淀 |
| "我们和竞品的技术架构对比" | 混合检索 | 需要内部架构（知识库）+ 竞品信息（外部） |
| "上周的故障报告在哪" | 知识库检索 | 内部历史文档 |

### 外部搜索结果处理

如果路由到外部搜索，结果需要处理后返回：

**可信度评估**：对搜索结果进行可信度判断

**与内部知识融合**：如果有相关内部知识，一并返回

**来源标注**：明确标注信息来源

## 多路召回

单一检索方式无法覆盖所有场景，需要多路召回。

### 召回通道

**文本召回**：
- 适用：精确关键词查询、专有名词查询
- 特点：高精确、低召回
- 索引：倒排索引

**向量召回**：
- 适用：语义匹配、概念查询
- 特点：高召回、需排序
- 索引：向量数据库

**图召回**：
- 适用：关联查询、因果追溯
- 特点：发现隐含关系
- 索引：图数据库

**摘要召回**：
- 适用：长文档定位
- 特点：快速定位文档
- 索引：摘要向量索引

### 召回策略

不同查询适用不同的召回组合：

**策略一：向量为主，文本为辅**

适用：概念查询、探索性查询

召回：
- 向量召回 Top K
- 文本召回补充
- 合并去重

**策略二：文本为主，向量为辅**

适用：精确查询、专有名词查询

召回：
- 文本召回 Top K
- 向量召回补充
- 合并去重

**策略三：多路并重**

适用：复杂查询、不确定意图

召回：
- 文本、向量、图并行召回
- 融合排序

### 召回数量控制

召回数量影响效率和准确性：

| 阶段 | 数量 | 说明 |
|-----|-----|-----|
| 初始召回 | Top 100-200 | 多路召回总和 |
| 粗排后 | Top 20-50 | 粗排过滤 |
| 精排后 | Top 5-10 | 最终返回 |

## 结果融合与排序

多路召回的结果需要融合和排序。

### 去重

同一文档可能被多路召回，需要去重：

- 按文档 ID 去重
- 保留最高分数
- 记录召回来源（哪些通道召回了它）

### 分数归一化

不同召回通道的分数不可比，需要归一化：

- 文本分数：BM25 分数
- 向量分数：相似度分数
- 图分数：关联强度

归一化方法：将各通道分数映射到相同区间，如 0-1。

### 融合策略

**加权融合**：

各通道分数按权重加权求和：

```
最终分数 = w1 × 文本分数 + w2 × 向量分数 + w3 × 图分数
```

权重根据查询类型动态调整，由 LLM 决定。

**RRF 融合**（Reciprocal Rank Fusion）：

基于排序位置而非绝对分数：

```
RRF分数 = Σ (1 / (k + 排名))
```

k 为常数，通常取 60。

优点：不受分数尺度影响，更稳健。

### 重排序

召回后的结果需要精排：

**特征维度**：
- 查询-文档相似度
- 文档质量（权威性、完整性）
- 时效性（创建/修改时间）
- 用户偏好（历史行为）

**重排序方式**：

方案一：使用 LLM 对 Top 结果进行相关性打分

方案二：训练排序模型（如有数据）

方案三：使用预训练的重排序模型

## 元数据过滤

元数据可以缩小检索范围，提高效率。

### 过滤时机

**检索前过滤**：
- 先按元数据筛选候选集
- 再在候选集中检索
- 适合过滤条件明确的查询

**检索后过滤**：
- 先检索召回
- 再按元数据过滤
- 适合过滤条件不明确的查询

### 常见过滤条件

**时间过滤**：
- 创建时间范围
- 修改时间范围
- "最近一周"、"去年"等

**类型过滤**：
- 文档类型（技术文档、会议纪要等）
- 文件格式（PDF、Word 等）

**来源过滤**：
- 作者
- 项目
- 部门

### 元数据与检索的结合

元数据过滤可以与检索并行：

```
用户查询 + 元数据条件
        │
        ▼
    向量检索 + 元数据过滤（向量数据库支持）
        │
        ▼
    返回满足条件的结果
```

大多数向量数据库支持在向量检索时同时进行元数据过滤。

## 检索策略选择

不同场景需要不同的检索策略。

### 查询驱动的策略选择

根据查询特点自动选择策略：

**方案**：

让 LLM 分析查询，判断适合的检索策略。输出策略配置，包括：
- 召回通道及权重
- 是否需要元数据过滤
- 召回数量

不预定义策略规则，由 LLM 根据查询自适应判断。

### 典型场景

**场景一：故障排查**

查询特点：问题描述、错误信息

策略：
- 文本召回：匹配错误码、关键词
- 向量召回：匹配问题描述
- 经验卡片索引：优先召回

**场景二：概念学习**

查询特点：宽泛概念、学习意图

策略：
- 向量召回为主
- 摘要召回辅助定位
- 返回完整文档

**场景三：精确查找**

查询特点：特定对象、已知信息

策略：
- 文本召回为主
- 元数据过滤缩小范围
- 高精度返回

**场景四：关联探索**

查询特点：多实体、关系查询

策略：
- 图召回：发现实体关系
- 向量召回：补充相关内容

## 检索增强

在基础检索之上，可以增加增强能力。

### 查询扩展

对用户查询进行扩展，提高召回：

**同义词扩展**：
- 查询"DB"扩展为"DB OR 数据库"
- 同义词来源于领域词库或 LLM 生成

**相关概念扩展**：
- 查询"微服务"扩展关联概念"服务网格"、"API网关"
- 由 LLM 判断需要扩展的概念

### 上下文增强

结合用户上下文优化检索：

**用户历史**：
- 用户查询过的内容
- 用户浏览过的文档
- 用户的行为偏好

**会话上下文**：
- 同一会话中的前序查询
- 前序查询限定的范围

**上下文利用方式**：

不硬编码规则，由 LLM 根据上下文信息调整检索策略。

### 混合检索

结合多种检索方式的优势：

**向量 + 关键词**：
- 向量召回语义相关内容
- 关键词召回精确匹配内容
- 融合排序

**稀疏 + 稠密向量**：
- 稀疏向量：保留词汇信息
- 稠密向量：捕获语义信息
- 混合索引和检索

### 反馈优化

利用用户反馈持续优化检索：

**显式反馈**：
- 用户点赞/踩
- 用户选择的结果
- 用户评分

**隐式反馈**：
- 用户点击的结果
- 用户停留时间
- 用户是否继续查询

**优化方式**：

收集反馈数据，用于：
- 调整排序权重
- 优化检索策略
- 训练个性化模型

## LLM优先与演进路径

### 当前方案的出发点

本文档默认的前提是：团队希望尽快搭出可用的Agent检索能力，但没有足够资源去训练专门的查询理解模型、路由模型、改写模型和排序模型。在这种前提下，采用 LLM first 的检索架构，是更泛化、也更经济实用的做法。

对于很多团队来说，当前阶段更重要的不是先把检索系统做到最优，而是先让 Agent 能够稳定回答大部分常见问题，尽快形成可用闭环。

### LLM first, but not LLM only

LLM 优先并不意味着所有检索环节都必须完全依赖 LLM。更合理的理解是：让 LLM 负责高语义、难枚举的理解和策略判断，让检索系统负责稳定执行和结果回退。

具体来说：
- LLM 更适合承担查询理解、查询路由、Query 分析、Query 改写和检索策略选择等高语义任务
- 文本召回、向量召回、图召回、元数据过滤、融合排序等执行层能力仍由确定性检索系统完成
- 系统可以对 LLM 输出做结构化约束，只允许其在已有信息源、召回通道、过滤条件和策略空间内选择
- 当 LLM 判断不明确、输出异常或策略效果不佳时，可以回退到通用混合检索路径

### 有专业算法团队时的增强方向

如果后续有专业的检索算法团队，可以在保留整体架构的前提下，逐步增加更可控的能力：
- 更稳定的内容理解与查询分类
- 更可控的 Query 分析和 Query 改写
- 轻量级的路由与召回策略模型
- 更细粒度的重排序、结果校正和反馈优化能力
- 更系统的离线评测、线上观测和效果迭代体系

### 关于评估体系

完整的准召率评测、离线标注集和 A/B 实验体系，对于追求效果上限的专业团队当然非常重要。但很多时候，团队的首要目标不是做出最优检索系统，而是快速搭出一个可用的 Agent。

在这种情况下，更现实的顺序通常是：
- 先把检索链路搭完整
- 先让 Agent 在主要场景下可用
- 再根据资源情况逐步补充评测、观测和专项优化能力

因此，本文档优先讨论可泛化落地的检索架构，不把复杂评测体系作为前置条件。

### 演进原则

推荐的演进方式是：
- 第一阶段：以 LLM 优先驱动查询理解、路由和改写，快速上线可用系统
- 第二阶段：通过日志和用户反馈识别高频查询模式
- 第三阶段：对高频稳定场景逐步引入规则、轻量模型或专项算法
- 第四阶段：形成“专用能力处理高频核心场景，LLM 处理长尾泛化场景”的混合体系

## 总结

### 检索流程要点

| 环节 | 关键决策 |
|-----|---------|
| 查询理解 | LLM 自适应理解，不预定义规则 |
| 召回策略 | 多路召回，权重动态调整 |
| 结果融合 | 分数归一化，加权或 RRF 融合 |
| 元数据过滤 | 检索前或检索中过滤 |
| 重排序 | 多维度特征，精排 Top 结果 |

### 泛化设计原则

整个检索方案遵循泛化原则：

- 不硬编码查询类型和策略规则
- 不预定义元数据过滤条件
- 由 LLM 根据查询自适应决策
- 通过反馈数据持续优化
